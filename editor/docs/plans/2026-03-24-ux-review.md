# UX Review: Hexerei Editor

**Date:** 2026-03-24
**Scope:** Comprehensive review of interaction patterns, visual identity execution, and discoverability. Starting from the issues captured in `editor/docs/ISSUES.md` and extending to cover the full user experience.

---

## Executive Summary

The editor has strong bones: the three-panel Spatial IDE layout works, HexPath-as-CLI is compelling, the sandtable visual identity document is well-conceived, and multi-geometry terrain is a differentiating feature. But the current implementation has gaps between the design vision and what's on screen. The biggest themes:

1. **Discoverability is poor** - new users have no way to learn what's possible
2. **Visual identity is partially applied** - some sandtable tokens are in place but several components use undefined/legacy CSS variables, and key design spec items (orientation thumbnails, hex-shaped palette chips, collapsible sections) aren't implemented
3. **Paint mode UX has friction** - entering/exiting paint mode, visual feedback during painting, and the relationship between terrain chips and paint actions isn't clear
4. **The NewMapDialog is disconnected** from the design system and has interaction bugs
5. **Feature Stack labeling and structure** needs rethinking for clarity

---

## 1. Onboarding & Discoverability

### Problem

A new user opening the editor has no guidance. There's no empty state, no tooltip system, no shortcut hints, no help overlay. The command bar modes (bare text = HexPath, `>` = command, `/` = search, `@` = goto) are invisible unless you already know them. Paint mode activation (clicking the tiny color chip vs. the terrain name) is especially obscure.

### Recommendations

**1a. Empty state / welcome screen**
When no map is loaded, instead of "Loading..." (which is a bug), show a welcome view in the canvas area:
- Large "Create New Map" button (triggers Cmd+N dialog)
- "Open Existing Map" button (triggers Cmd+O)
- Brief list of keyboard shortcuts (Cmd+K for command bar, Cmd+N, Cmd+O, Cmd+S)
- Maybe a one-line tagline: "Spatial IDE for hex maps"

**1b. Command bar placeholder text**
The command bar should show contextual placeholder text:
- No selection: `Type a HexPath, or > for commands, / to search, @ to jump...`
- Hex selected: `Type to add features at {hexLabel}, or > for commands...`
- Feature selected: placeholder could show the feature's current `at` expression

**1c. Keyboard shortcut overlay**
Add a `>shortcuts` command (or `?` key) that shows a modal/overlay listing all shortcuts. This is cheap to build and solves the "how do I..." problem.

**1d. Paint mode affordance**
The color chip click-to-paint interaction needs better visual signaling. Options:
- Add a small paintbrush icon overlay on hover for terrain chips
- Add a tooltip: "Click chip to paint, click name to edit"
- Consider a dedicated "Paint" button per terrain row instead of overloading the chip

**1e. Status bar hints**
The status bar has unused real estate. When idle, the center segment (currently just the map title) could rotate through contextual tips: "Cmd+K: Command bar", "Click hex to inspect", "Click color chip to paint".

## 2. Visual Identity: Sandtable Gaps

The sandtable theme CSS (`theme-sandtable.css`) is well-defined with the right tokens. But several components aren't consuming them correctly.

### 2a. Undefined CSS variables (bug)

`NewMapDialog.css` uses variables that don't exist in any theme file:
- `--bg-panel` (should be `--bg-surface` or `--bg-elevated`)
- `--bg-input` (should be `--bg-base` or `--bg-surface`)
- `--bg-hover` (should be `--bg-elevated`)
- `--border-color` (should be `--border-subtle` or `--border-focus`)

`Inspector.css` also uses `--bg-panel` for the paint-active chip ring.

This means the dialog and parts of the inspector are rendering with browser defaults (transparent/black), which explains the user feedback that the dialog "doesn't seem visually distinguished" and "style doesn't seem to match."

**Fix:** Replace all undefined variables with their design-system equivalents.

### 2b. Dialog styling

Beyond the broken variables, the NewMapDialog needs sandtable treatment:
- Background should use `--bg-elevated` with a `--border-accent` (teal) border
- The overlay backdrop should be darker (`rgba(0, 0, 0, 0.7)`) to push focus
- Title should use the accent-colored heading style matching panel headers
- Form inputs should have `--border-focus` borders and `--bg-surface` backgrounds
- The "Create" button glow should use `box-shadow: 0 0 8px var(--accent-hex-glow)`

### 2c. Inspector section headers

The inline styles `style={{ padding: '0 0 8px 0', marginBottom: '12px', fontSize: '10px' }}` are scattered across every `<h3 className="inspector-header">` in Inspector.tsx. These should use the `.inspector-header` class consistently (which already has the right sandtable styling in CSS), and the inline overrides should be removed or moved to CSS. The repetition suggests the CSS class isn't being applied correctly alongside these overrides.

### 2d. Feature Stack "+" button

The `+` button in the feature stack header reads as part of the "FEATURE STACK" label ("FEATURE STACK+"). It needs:
- Visual separation: margin-left auto, distinct button styling
- Different color or a border to read as a control, not part of the heading text
- Tooltip: "Add empty feature (Cmd+Enter)" or similar

### 2e. Hex labels on canvas

User reports labels are "almost unreadable blurry." The sandtable spec calls for `rgba(0, 180, 220, 0.35)` with a subtle `0 0 4px rgba(0, 180, 220, 0.15)` glow shadow. Need to verify:
- Is canvas rendering at device pixel ratio? (If not, labels will be blurry on retina)
- Is the glow shadow applied? (shadowBlur on canvas text can cause blur if overused)
- Consider making label opacity scale with zoom (fainter when zoomed out, brighter when zoomed in)

### 2f. Favicon

No favicon exists. The identity calls for a tactical/holographic feel. Suggestion: a single hexagon outline in the accent cyan (`#00D4FF`) on transparent background. Simple, on-brand, works at 16x16 and 32x32.

## 3. Terrain Palette & Paint Mode

### 3a. Palette naming

"Starter Palette" in the NewMapDialog should be "Terrain Palette" — it's setting the initial terrain vocabulary, not a temporary starting point. The user explicitly called this out.

### 3b. Palette chip design (from ISSUES.md)

The current terrain chips are 12x12px colored squares. The user asked: can we show hex-shaped chips (respecting flat/pointy orientation), styled per geometry type (filled for hex terrain, line for edge terrain, dot for vertex terrain)?

This would:
- Reinforce the geometry-type distinction (hex/edge/vertex) visually
- Look distinctive and on-brand
- Provide visual consistency between palette and canvas

For the feature stack specifically: instead of the square color chip + terrain name, show a small hex chip + feature name (not terrain name, since a feature IS a named thing, and terrain is a property of it).

### 3c. Terrain section grouping

The three terrain sections (HEX TERRAIN, EDGE TERRAIN, VERTEX TERRAIN) look like completely separate panels. They should be visually grouped as sub-sections of a single "TERRAIN" section. Options:
- Tabbed interface: three tabs (Hex | Edge | Vertex) with geometry-typed accent colors
- Collapsible sub-sections under a single "TERRAIN VOCABULARY" heading
- A segmented control at the top choosing which geometry type to display

The tabbed approach is probably best for space efficiency and aligns with the geometry-typed color system (cyan tab for hex, magenta for edge, yellow for vertex).

### 3d. Paint mode entry/exit

Current: click color chip → paint mode. Click chip again or Esc → exit.

Issues:
- No visual confirmation of entering paint mode besides the status bar message and cursor change
- Clicking "off the map" doesn't exit paint mode (user expected it would)
- The relationship between "select a terrain chip to paint" and "click the terrain row to edit" is confusing — same row, different click targets with very different effects

Proposal:
- Paint mode activation should flash/pulse the canvas border briefly (or the status bar paint segment) as a visual confirmation
- Add a "Paint Mode" floating badge near the canvas (top-left corner?) with terrain color, name, and an "X" to exit
- Clicking anywhere in the canvas area that's off-map should exit paint mode (or at minimum, not crash — see the shift-click VOID bug)
- Single-click terrain chip → paint mode, double-click → edit properties (since editing is much less common than painting)
- This is cleaner than a separate paint button and keeps the chip as the primary interaction point

### 3e. HexPath bar sync during painting

User notes the command bar doesn't update as terrain is painted on the map. When painting adds/modifies features, the command bar should reflect the selection or show the HexPath of the most recently painted feature. This maintains the "DSL as source of truth" principle.

**HexPath display format:** Use comma separators for readability: `a2-a6, b9, c3` rather than `a2-a6 b9 c3`. The RFC recommends but doesn't require commas — the editor should use them as the canonical display format.

### 3e-bis. Feature selection highlight: segment-aware paths

When selecting a feature, the dotted-line highlight currently connects all hexes in the HexPath as a single path. It should be segment-aware: only draw dotted connecting lines within contiguous segments. For `a2-a6, b9, c3`: dotted line from a2 through a6, but b9 and c3 should just be outlined individually (no connecting line). This makes the HexPath structure visible in the highlight.

### 3f. Feature identity on repeated terrain

Painting forest, then something else, then more forest currently merges into the first forest feature. User expects separate features (forest1, forest2). This is a data model question but has UX implications: if features auto-merge, the user loses the ability to style/edit regions independently. Each paint session should create a new feature.

**Future:** Add a "merge features" action (select multiple features in stack → right-click or command → merge into one). This is the inverse — easy to merge later, hard to split.

## 4. Layout & Inspector Improvements

### 4a. Collapsible sections

Inspector sections should be collapsible (user request). Each section header (`MAP METADATA`, `LAYOUT`, `TERRAIN`, `FEATURE PROPERTIES`, `GEOMETRY`, etc.) should be a disclosure triangle. Default state: all open. Collapsed state persists per section.

This is especially important as the inspector gets longer with hex/edge/vertex terrain sections.

### 4b. Orientation selector

The orientation dropdown (`flat-down`, `flat-up`, `pointy-right`, `pointy-left`) should be replaced with a 2x2 button grid showing thumbnail hex images for each orientation. Each button shows a small hex cluster in the correct orientation. This is faster to use and more intuitive than parsing text labels.

Similarly, origin should be a 2x2 grid of buttons showing the origin position with an arrow or dot.

Both should be available in the Layout section of the inspector (currently only in NewMapDialog).

### 4c. Base Layer in the stack

User asks: does "Base Layer" need to be separate from the stack? The current design pins it at the bottom with `order: 9999`. The name "Base Layer" is also odd — it's really just the first feature, typically with `at: "@all"`.

Proposal: show it as the first item in the stack (it's index 0) without special treatment. Label it the same way as other features — show its `at` expression (`@all`), terrain name, etc. No special "Base Layer" text. A subtle "(base)" badge is fine but the primary label should follow the same pattern as every other feature. Remove the visual separation and pinning. The stack's ordering should be self-explanatory.

### 4d. Inspector values: label vs. data contrast

The sandtable spec says read-only data values should use `--text-primary` (bright) while labels stay `--text-secondary`. Check that this is consistently applied. In the hex/edge/vertex inspector views, the terrain name, coordinate, etc. should be bright text.

**Feature stack path contrast:** The `at` expression text in the feature stack (currently `rgba(0, 180, 220, 0.5)`) is too dim for path features where the `at` string is the most important identifying information. Bump to at least `0.7` opacity, or use `--text-secondary` as a floor.

## 5. Canvas & Viewport

### 5a. Zoom performance and display

User reports zoom feels slow. Questions to investigate:
- Is the canvas rendering at full resolution on every frame, or is there a fixed-resolution bitmap being scaled?
- Does the renderer only draw visible hexes, or the entire mesh?
- On zoom, does the canvas re-render immediately or is there debouncing?

The zoom value in the status bar shows excessive decimal precision (e.g., "127.3847%"). Should be rounded to integer percentage.

**Scope:** Zoom performance is potentially a deeper architectural issue (rendering strategy, viewport culling, etc.). Investigate first, then decide on scope — this may warrant its own phase separate from the UX polish work.

### 5b. Aspect ratio on resize

User reports window resizing squashes/stretches the map. This means the canvas element is being CSS-stretched without updating its internal dimensions. The canvas `width`/`height` attributes must be updated on resize (via ResizeObserver or window resize event), and the viewport recalculated to maintain proper aspect ratio.

### 5c. Edge/vertex selection in pointy orientations

User confirms edge and vertex hit-testing is wrong in pointy-left and pointy-right orientations (hex selection works). This needs a test covering all four orientations for each geometry type, and likely a coordinate transform fix in the hit-test code.

### 5d. Linear hex terrain rendering

User notes that linear (path) hex terrain is rendered with both fill AND the path line. It should only have the path line (so it can visually stack with other terrain), and the path should be thicker for visibility.

## 6. NewMapDialog

### 6a. Loading state bug

When the NewMapDialog is visible, "Loading..." appears behind it. Canceling the dialog leaves the app stuck in the loading state with no way to recover. The `Loading...` text comes from `Inspector.tsx` line 33 when `model` is null. After canceling, the app needs to either:
- Return to the previous model state (if one existed)
- Show the empty/welcome state (if no map was loaded)

### 6b. first_col / first_row

User notes that first col/row "is not part of the RFC" and wants to brainstorm how `@all` vs `rows x cols + first` should work with orientation. This is a data model question but the dialog exposes it as raw numeric inputs, which is confusing. Options:
- Remove first_col/first_row from the dialog entirely (default to 1,1)
- Show a preview of the resulting hex labels as the user adjusts settings
- Group first_col, first_row, and label_format into a "Numbering" section with a live preview

**Deferred:** The first_col/first_row interaction with @all, orientation, and the RFC needs a separate design discussion. Remove from the initial UX phases; follow-on work.

### 6c. Dialog visual hierarchy

The dialog is a flat list of fields. Group them:
1. **Map identity**: Title (currently hardcoded to "New Map" — should be editable in dialog)
2. **Grid geometry**: Width, Height, Orientation (with thumbnail picker), Origin (with thumbnail picker)
3. **Labeling**: Label format, First column, First row (with live preview showing a sample label)
4. **Terrain**: Palette selector, Base terrain

## 7. Interaction Consistency

### 7a. Escape key behavior

Escape currently chains: exit paint mode → clear selection → unfocus command bar. This is reasonable but could be clearer. Consider:
- First Escape: exit paint mode (if active)
- Second Escape: deselect (if anything selected)
- Third Escape: unfocus command bar (if focused)

Currently all three happen in a single press, which can be disorienting.

### 7b. Click semantics on canvas

Canvas click currently does: if paint mode → paint; else → hit-test and select. But:
- Clicking off-map (void space) in paint mode crashes (VOID hex bug)
- Shift+click for edge path drawing crashes
- There's no click-to-deselect (clicking empty canvas space should deselect)

### 7c. Drag-to-reorder features

The vision doc mentions drag-to-reorder in the feature stack. The drag handle is commented out in the code (`{/* <div className="feature-drag-handle">⋮⋮</div> */}`). This is important for the "features override in stack order" model. Should be prioritized.

## 8. Proposed Implementation Order

Organized by impact and independence. Each group can be a separate PR.

### Phase A: Bug Fixes (foundation)
1. Fix undefined CSS variables in NewMapDialog.css and Inspector.css
2. Fix "Loading..." state when dialog is canceled
3. Fix shift-click VOID crash (edge path drawing)
4. Fix edge/vertex hit-testing in pointy orientations
5. Fix canvas aspect ratio on window resize
6. Round zoom percentage to integer in status bar

### Phase B: Visual Identity Completion
1. Restyle NewMapDialog with sandtable design system
2. Remove inline style overrides from Inspector section headers
3. Add favicon (hex outline in cyan)
4. Verify canvas DPR rendering for label crispness
5. Fix linear hex terrain rendering (path only, no fill)

### Phase C: Discoverability
1. Empty state / welcome screen when no map loaded
2. Command bar contextual placeholder text
3. `>shortcuts` command or `?` key overlay
4. Paint mode floating badge with exit button
5. Tooltip on terrain color chip ("Click to paint")
6. Feature stack "+" button visual separation and tooltip

### Phase D: Inspector & Palette UX
1. Collapsible inspector sections
2. Orientation thumbnail button bar (inspector + dialog)
3. Origin thumbnail button bar (inspector + dialog)
4. Tabbed terrain sections (Hex | Edge | Vertex)
5. Terrain palette rename ("Starter Palette" → "Terrain Palette")
6. Base layer: remove special treatment, show as first stack item

### Phase E: Paint & Authoring Flow
1. Separate features per paint session (no auto-merge)
2. Command bar sync during painting (with comma-separated HexPath display)
3. Off-map click safety (no crash, exit paint mode)
4. Paint mode visual confirmation (flash/pulse)
5. Single-click chip → paint, double-click → edit terrain properties
6. Segment-aware feature selection highlights (dotted lines only within contiguous segments)

### Phase F: Canvas & Performance
1. Zoom performance investigation (may become its own phase depending on findings)
2. Canvas DPR-aware rendering
3. Hex-shaped terrain chips in palette and feature stack
4. Canvas hover glow (hex-under-cursor)
5. Drag-to-reorder features in stack

### Phase G: Dialog Improvements
1. Editable title in NewMapDialog
2. Field grouping (identity / geometry / labeling / terrain)
3. Live label preview in dialog
4. ~~first_col/first_row~~ — deferred to separate design discussion
