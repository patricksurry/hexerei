  3. type → modifier boolean (04-data-model.md:206)

  The current type: "base" | "modifier" is a string enum with default "base". Replacing with
  modifier: boolean (default false) is simpler and avoids a field called type (which is a
  reserved/overloaded word in many languages). The schema, inline schema, and prose all need
  updating.

  4. path boolean on terrain types (04-data-model.md:208)

  This is a significant semantic addition. It would distinguish "road" (path-like, connecting hex
  centers) from "forest" (area-like). The appendix A note at line 73 confirms the intent: linear
  features represented as hexes should be tagged path: true. This needs:
  - Adding path to the TerrainTypeDef in both schemas
  - Defining the semantic implications in the data model (e.g., rendering should connect
  through-hex segments, adjacency/connectivity is segment-based not area-based)
  - Clarifying the interaction with edges: your comment notes edge→path is essentially an edge
  sequence already, so path mainly matters for hex terrain

  For clarity, only hexpath segments of length > 1 contribute to path: true terrain, singletons are ignored.

  5. YAML as primary format (08-serialization.md:12)

  The introduction (01-introduction.md:12) explicitly says "YAML is the primary authoring format."
  But Section 8 leads with "The canonical serialization is JSON" and recommends .hexmap.json. The
  tension is intentional (JSON for machine interchange, YAML for authoring), but the section
  ordering and emphasis make it feel JSON-first. Consider either:
  - Reordering Section 8 to lead with YAML (matching the intro's emphasis)
  - Or adding a sentence at the top clarifying the dual-format intent

  7. Vertex terrain examples (appendix-a-terrain.md:52)

  Additional Inconsistencies & Underspecified Areas

  A. side field exists in schema but not in prose

  Both rfc/hexmap.schema.json:127-129 and the inline schema define a side field on
  FeatureAttributes with values ["both", "in", "out", "left", "right"]. But Section 4 (Features)
  never mentions side at all. The onesided flag on terrain types is described, but there's no
  explanation of how a feature specifies which side it's on. This is a gap — side needs to be
  documented in the Feature Attributes table.

[what's your proposal here?  i don't like the current ambiguity in schema between in/out/left/right either.
some analogue of right-hand rule could work for individual edges since you can address them from two sides,
but is that consistent with a path of edges since you'd want same sideness along the path?]

  B. terrain pattern in schema is too restrictive for multi-terrain

  The prose (04-data-model.md:260) says terrain is "One or more space-separated terrain types." But
   the schema pattern is ^[a-z][a-z0-9_]*$ — which rejects spaces. So terrain: "forest hill"
  (modifier stacking) would fail validation. The regex should be something like ^[a-z][a-z0-9_]*(
  [a-z][a-z0-9_]*)*$.

  C. metadata.additionalProperties: false contradicts the prose

  Section 4 (04-data-model.md:38) says "Implementations MUST preserve unrecognized fields in
  metadata and pass them through unchanged." But the schema sets additionalProperties: false on
  metadata, which means any extra fields will fail validation. One of these needs to change.

[iirc this was a tweak for ajv validation, but sounds like we fixed it wrong, the doc is correct.  let's add a test]

  D. Regions section references hexes but the key is at

  04-data-model.md:337 says "A region is simply a feature with an id and a hexes collection." But
  features use the at key, not hexes. The example right below correctly shows at:, so the prose is
  just wrong.

  E. Test cases reference "Appendix D, Q1" but appendices only go to C

  12-test-cases.md:31 says "Multi-board composition deferred to v2 (see Appendix D, Q1)." There is

  F. hex_top used in addressing diagrams but not defined as a field

  05-addressing.md:67 and :86 reference hex_top: flat and hex_top: pointy, but the actual layout
  field is orientation with values flat-down/flat-up/pointy-right/pointy-left. The diagrams should
  use the actual field names. (this is legacy terminology that wasn't updated)

  G. origin effect on coordinate conversions is unspecified

  Section 7 (geometry) defines offset→cube conversions assuming top-left origin with standard
  numbering direction. But origin can be bottom-left, top-right, or bottom-right — which flip
  row/column numbering directions. Section 7 never explains how origin affects the (col, row)
  values fed into the conversion formulas. Does the implementation negate/mirror the indices before
   conversion? This needs to be specified.

  I. Edge feature examples inconsistent with HexPath connectivity

  04-data-model.md:318 shows at: "0304/N 0304/NE 0404/N" described as "A path of three edges." But
